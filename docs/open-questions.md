# 미결 질문

이 문서는 MVP 도메인 모델과 DB 스키마 초안에서 최종 결정이 필요한 질문을 모은다.
제품/도메인 동작이 애매한 부분은 구현에서 임의로 확정하지 않는다.

## 복습 일정

1. `nextReviewAt`의 날짜 기준은 무엇인가?
   - 추천안: 사용자 기준 로컬 날짜를 사용한다.
   - 대안: 서버 시간대 기준 날짜를 사용한다.
   - 트레이드오프: 사용자 로컬 날짜는 체감 UX가 좋지만 사용자 시간대 설정이 필요하다. 서버 기준은 단순하지만 날짜 경계가 사용자의 기대와 다를 수 있다.

2. 문제를 등록했지만 아직 풀이 시도가 없는 경우 `nextReviewAt`은 어떻게 둘 것인가?
   - 추천안: `null`로 둔다.
   - 대안: 등록일 또는 다음 날로 둔다.
   - 결정 필요: 등록만 한 문제도 복습 큐에 노출할지 정해야 한다.

3. `MASTERED` 처리는 수동인가, 자동인가?
   - 추천안: MVP에서는 수동 처리로 둔다.
   - 대안: 연속 성공 횟수를 기준으로 자동 처리한다.
   - 결정 필요: 자동 처리라면 필요한 성공 횟수와 성공 기준을 정해야 한다.

4. 복습 성공이 여러 번 이어질 때 간격을 언제, 어떻게 늘릴 것인가?
   - 추천안: MVP에서는 고정 간격만 사용한다.
   - 대안: 성공 횟수 기반으로 간격을 점진 증가시킨다.
   - 결정 필요: 증가 공식은 별도 도메인 규칙으로 확정해야 한다.

## 중복 문제 등록

5. 중복 기준은 `platform + externalProblemId`로 확정할 것인가?
   - 추천안: `platform + externalProblemId`를 유니크 기준으로 사용한다.
   - 대안: 정규화된 URL을 기준으로 사용한다.
   - 결정 필요: 외부 문제 ID가 없는 플랫폼을 지원할 계획이 있는지 확인해야 한다.

6. 같은 문제가 여러 플랫폼에 존재하는 경우 같은 문제로 묶을 것인가?
   - 추천안: MVP에서는 서로 다른 문제로 본다.
   - 대안: 별도 canonical problem 개념으로 묶는다.
   - 결정 필요: 크로스 플랫폼 통합은 MVP 범위인지 확인해야 한다.

## 태그

7. 태그 이름은 대소문자를 구분할 것인가?
   - 추천안: 저장 전 trim하고 대소문자는 표시값으로 유지하되, 중복 비교 정책은 별도로 정한다.
   - 대안: 모든 태그를 대문자 또는 소문자로 정규화한다.
   - 결정 필요: `DP`와 `dp`, `다이나믹 프로그래밍`을 같은 태그로 볼지 정해야 한다.

8. 태그 별칭을 지원할 것인가?
   - 추천안: MVP에서는 지원하지 않는다.
   - 대안: `tags` 마스터 테이블과 alias 테이블을 둔다.
   - 결정 필요: 취약 태그 통계에서 별칭을 합산해야 하는지 정해야 한다.

9. 태그는 문제 등록 시 사용자가 직접 입력하는가, 플랫폼 태그를 수동으로 옮겨 적는가?
   - 추천안: MVP에서는 사용자가 직접 입력한다.
   - 대안: 플랫폼별 태그 목록을 사전에 정의한다.
   - 결정 필요: 문제 크롤링이 비목표이므로 자동 태그 수집은 제외하는 것이 맞는지 확인해야 한다.

## 상태와 enum

10. 플랫폼 enum에 `ETC`를 둘 것인가, `OTHER`를 둘 것인가?
    - 추천안: `ETC`를 둔다.
    - 대안: `OTHER`를 둔다.
    - 결정 필요: API 문서와 화면 표기 기준에 맞춰 하나로 확정해야 한다.

11. `RETRY_NEEDED`와 `FAILED`의 구분 기준은 무엇인가?
    - 추천안: `FAILED`는 실패 종료, `RETRY_NEEDED`는 풀었거나 풀다 멈췄지만 다시 풀기로 표시한 상태로 둔다.
    - 대안: `RETRY_NEEDED`를 제거하고 `FAILED`와 메모로만 표현한다.
    - 결정 필요: 복습 간격이 다르므로 사용자가 어떤 기준으로 선택할지 설명이 필요하다.

12. `SOLVED_WITH_HINT`의 힌트 범위는 어디까지인가?
    - 추천안: 공식 풀이, 블로그, AI, 친구 도움 등 외부 힌트를 본 경우로 넓게 정의한다.
    - 대안: 힌트 유형을 별도 필드로 나눈다.
    - 결정 필요: 취약 태그 통계에서 힌트 사용을 어떻게 해석할지 정해야 한다.

## 스키마 세부사항

13. `difficulty`는 자유 문자열로 둘 것인가, 플랫폼별 정규화 모델을 둘 것인가?
    - 추천안: MVP에서는 자유 문자열로 둔다.
    - 대안: 플랫폼별 난이도 체계를 별도 컬럼 또는 enum으로 모델링한다.
    - 결정 필요: 난이도별 통계나 필터가 MVP에 필요한지 확인해야 한다.

14. `mistakeReason`은 자유 텍스트인가, enum인가?
    - 추천안: MVP에서는 자유 텍스트로 둔다.
    - 대안: 자주 쓰는 실수 유형 enum과 자유 메모를 분리한다.
    - 결정 필요: 실수 유형 통계가 MVP에 필요한지 확인해야 한다.

15. 문제 삭제를 허용할 것인가?
    - 추천안: MVP에서는 실제 삭제보다 `MASTERED` 또는 향후 `ARCHIVED` 상태를 우선 고려한다.
    - 대안: 문제 삭제 시 풀이 시도와 태그를 cascade 삭제한다.
    - 결정 필요: 학습 이력 보존이 중요한지, 잘못 등록한 문제 삭제가 더 중요한지 정해야 한다.

16. 향후 인증을 도입하면 유니크 기준에 `user_id`를 포함할 것인가?
    - 추천안: MVP에서는 인증이 비범위이므로 단일 사용자 기준으로 모델링한다.
    - 대안: 처음부터 `user_id`를 모든 주요 테이블에 포함한다.
    - 결정 필요: 가까운 시점에 다중 사용자 전환 계획이 있는지 확인해야 한다.

## 취약 태그 통계

17. `weaknessScore` 공식은 어떻게 정의할 것인가?
    - 추천안: MVP에서는 `failedCount`와 `hintUsedCount`를 기반으로 단순 계산한다.
    - 대안: 최근 시도에 가중치를 주거나 풀이 시간까지 반영한다.
    - 결정 필요: 점수 공식은 사용자에게 설명 가능한 방식이어야 한다.

18. 취약 태그 통계는 `MASTERED` 문제의 시도도 포함할 것인가?
    - 추천안: MVP에서는 전체 풀이 이력을 포함한다.
    - 대안: 현재 `ACTIVE` 문제만 포함한다.
    - 결정 필요: 통계가 장기 약점 분석인지, 현재 복습 대상 분석인지 정해야 한다.
